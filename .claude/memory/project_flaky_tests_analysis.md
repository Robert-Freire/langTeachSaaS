

### 3. `teacher-profile.spec.ts` — "teacher can save and reload profile settings"

**Symptom:** After saving profile and reloading, `displayName` shows `e2e-test@langteach.dev` instead of `Test Teacher`. Sometimes the `save-success` toast never appears at all.

**Root cause (preliminary, 2026-03-15):**
- Passes in isolation, fails when running full suite with 8 parallel workers
- All tests share the same Auth0 test user (`E2E_TEST_EMAIL`), so parallel logins cause Auth0 session/token conflicts
- Two distinct failure modes observed:
  1. `save-success` toast never appears (save API call likely fails due to stale/conflicting auth token)
  2. After reload, `displayName` shows email instead of saved value (profile data not persisted, or a parallel test's upsert overwrites the saved display name)
- The Settings page has a loading gate (`isLoading` check), but the `useEffect` that populates `displayName` from the profile runs one render cycle after loading completes, creating a brief window where the input has the initial empty value
- The `UpsertTeacherAsync` backfill logic (ProfileService lines 86-88) should NOT overwrite "Test Teacher" since it contains a space and `LooksAutoGenerated` returns false for names with spaces

**Partial fix applied:** Changed `waitForSelector` + immediate `toHaveValue` to `toHaveValue` with 10s timeout. This addresses the render-cycle timing issue but NOT the parallel test auth interference.

**Remaining fix options:**
1. Isolate this test from parallel execution (`test.describe.serial` or separate project in playwright config)
2. Create a dedicated Auth0 test user for this test to avoid token conflicts
3. Add `test.setTimeout` and retry logic for the save step
